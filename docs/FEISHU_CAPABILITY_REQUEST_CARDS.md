# Feishu Capability Request Cards

TONA can ask for a Feishu-related capability from inside a chat. It always explains the purpose, relevant scope, and boundary before a user confirms.

## What a user can say to a bot

Mention the target bot, then send one of the following:

- \`申请权限：读取群消息作为知识上下文\`
- \`申请权限：读取飞书文档\`
- \`申请权限：创建飞书文档\`
- \`申请权限：读取日历\`
- \`申请权限：创建日程\`
- \`申请技能：联网检索\`

TONA sends an interactive card with **Confirm** and **Not now** buttons. Only the user who made the request can confirm it.

## One-time Feishu setup

For each Feishu application that should send these cards:

1. In **Events and Callbacks**, configure the normal event subscription URL using the workspace-specific URL generated by TONA:
   \`https://tona-agent-studio.onrender.com/feishu/events/<workspaceId>\`.
2. In the **Callback Configuration** tab, subscribe to the card callback event \`card.action.trigger\`.
3. Use the same workspace-specific URL as the callback request URL.
4. Save and publish the Feishu app.

The callback is answered immediately with a Feishu toast notification.

## Approval outcomes

| Request type | What the card confirms | What still has to happen |
| --- | --- | --- |
| App permission, such as \`im:message.group_msg\` or document write | The request is recorded | An app administrator must open the scope in Feishu Open Platform and publish the app |
| Personal data, such as documents or calendar | The user’s authorization intent is recorded | OAuth redirect and the requested scopes must be configured before a personal authorization page can be opened |
| Uninstalled skill | The request is recorded | TONA needs a real tool/skill executor before it can perform the work |

A card never bypasses Feishu permission review, never grants another person’s data access, and never performs write/delete/send operations merely because it was clicked.
